A design-system agent your engineering agents can call

Startups that let engineers "just build the UI" ship 14 shades of gray, 6 button styles, and an onboarding flow that feels like four different apps. Instead, build a single design agent that owns tokens, components, and brand — and expose it as an MCP server that every engineering agent must route through.

design system tokens Figma MCP accessibility brand

Why an agent, not just a style guide

A style guide is a PDF nobody reads. A design-system agent is a program that answers questions, rejects proposals, and emits code. Engineering agents get better outcomes because they can ask, not guess.

01

Define tokens before components

Tokens are the atom of your design system. Get these right and everything else cascades.

  • Color — semantic (bg, fg, accent, danger) not literal (blue-500).
  • Spacing — one scale (4, 8, 12, 16, 24, 32, 48, 64). No freelance 13px paddings.
  • Type — 5 ramps max, named by role (display, heading, body, mono, caption).
  • Radius, shadow, motion — small enumerated sets. "subtle / medium / dramatic".
Store tokens in one file (tokens.json or Style Dictionary). It's the source of truth that feeds Figma variables and your CSS variables and the agent.
02

Ship the component library once, reuse forever

Build ~25 primitives in code. Don't recreate per-feature.

  • Button, Input, Select, Checkbox, Radio, Switch, Textarea, DatePicker.
  • Card, Dialog, Drawer, Popover, Tooltip, Toast, Tabs.
  • Table, Pagination, Skeleton, EmptyState, ErrorBoundary.
  • Layout primitives: Stack, Cluster, Box, Grid.
Use shadcn/ui or Radix + Tailwind to shortcut this in ~3 days. Skin with your tokens. Done.
03

Build the Design Agent (MCP server)

A tiny TypeScript MCP server exposing three tools every engineering agent calls.

  • get_token(name) — returns the semantic value for a token.
  • get_component(name) — returns the current component API + a code example.
  • review_ui(diff) — takes a UI diff, returns violations ("used hex color outside tokens", "button not from DS").
# Engineering agent asking the design agent
> I need to add a destructive confirmation dialog.
  > tool_use: design.get_component("ConfirmDialog")
  > tool_use: design.get_token("color.danger.fg")
  <- returns component API + token value + a11y note
04

Figma MCP: close the loop with designers

When you have a part-time designer, connect Figma via MCP so agents can read frame specs and designers can export tokens.

  • Engineering agents pull frame JSON to understand intent.
  • Figma variables sync to tokens.json on commit — one source of truth.
  • Optional: a "figma-to-code" agent that produces a first-pass React component from a selected frame, wired to your tokens.
05

Brand voice is part of design

Microcopy is design. Store it with the system and make the design agent own it too.

  • Sentence case everywhere unless explicitly brand.
  • Error copy rules: name what happened, what's expected, what to do next. No "Oops!"
  • Empty-state rules: a line about what will appear, a primary action, no illustrations that mean nothing.
  • Confirmation dialogs: verb-first button labels ("Delete draft" beats "OK").
06

Accessibility baked in, not bolted on

Enterprise customers will ask. Ship accessible by default; refuse to expose components that aren't.

  • Every component renders with a visible focus ring using --focus token.
  • Contrast: AA by default; AAA for body copy where feasible.
  • CI eval: axe-core runs against every Storybook story; PRs fail on violations.
  • Keyboard map documented per component — agents reading the API see it.
07

Review gates for UI PRs

The design agent becomes a required reviewer in CI for any PR touching *.tsx or *.css.

  • Parses the diff, calls review_ui, comments inline on violations.
  • Blocking only for token violations, accessibility regressions, and new "one-off" components.
  • Non-blocking suggestions for microcopy and spacing rhythm.
Rule: if an engineer needs a component that doesn't exist, they open a design-system ticket — they don't ship a one-off. The agent enforces this.
08

Marketing & product share the system

Your landing pages and your app should feel like one company.

  • Marketing site imports the same token package as the product.
  • A subset of components (Button, Card, Section) are shared.
  • The marketing agent (see Marketing) calls the same get_token tool so ad creative stays on-brand.
