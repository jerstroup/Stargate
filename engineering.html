An agentic SDLC: MVP in weeks, enterprise-ready from day one

Most AI coding setups get you to a demo and then collapse under the weight of a real codebase. This is the opposite: a layered agentic system — planner, coder, reviewer, test, deploy, and oncall — that can take a three-person team from prototype to SOC2-ready production without rewriting everything at Series A.

Claude Code Agent SDK MCP CI evals monorepo

The target architecture

Six agents, one orchestrator, one eval harness. You won't build all of them day one — you'll build them in the order below.

// agents

  • Planner — turns a spec into a task DAG
  • Coder — implements one node at a time
  • Reviewer — critical PR feedback
  • Test author — writes + maintains tests
  • Release — runs migrations, canaries, rollback
  • Oncall — triages alerts, drafts incident notes

// shared substrate

  • Single monorepo (Turbo/Nx)
  • MCP servers for: db, search, logs, Linear, Figma
  • /CLAUDE.md + /skills/*.md in repo root
  • Eval suite in /evals
  • Trace store (Langfuse)

The rollout

Ten steps. Roughly 6-10 weeks elapsed for a 2-3 person team.

01

Make the repo agent-legible

Before any agent, invest 1 day writing a great CLAUDE.md. It is the single highest-leverage artifact in your codebase.

  • Architecture overview (5-10 bullets, not an essay).
  • "How we do X": testing, migrations, deploys, adding a new endpoint.
  • Non-obvious invariants ("we never mutate req.ctx after middleware").
  • Command cheatsheet: pnpm dev, pnpm test, pnpm db:migrate.
  • Links to the style guide, the design-system agent, and the data dictionary.
02

Pick one coding IDE & standardize

Heterogeneity costs you. Pick Claude Code as the default terminal agent — team installs, shares hooks, shares subagents. Everyone gets the same capabilities on day one.

  • Commit .claude/settings.json to the repo (permissions, hooks).
  • Commit .claude/agents/*.md — shared subagents like reviewer, test-writer, explorer.
  • Commit .claude/skills/*/SKILL.md — opinionated workflows (e.g. add-feature-flag, new-migration).
Team skills = institutional knowledge that compounds. Every time you solve a tricky workflow twice, skillify it.
03

Wire your MCP tool surface

The capabilities your agents share live here. Build five servers in this order:

  1. db — read-only SQL against staging. Never prod.
  2. logs — structured search over the last 7 days.
  3. linear (or similar) — tickets in, tickets out.
  4. figma — pull frame specs for UI tasks.
  5. docs — semantic search over /docs + Notion.
Credential hygiene: every MCP server runs with least-privilege tokens issued per-developer and rotated weekly. Agents never see prod write credentials.
04

Build the Planner agent

The Planner turns a spec (markdown from PM or founder) into a task DAG with acceptance criteria per node. It does not write code.

  • Input: a spec file + the repo context + the design tokens.
  • Output: plan.md with a numbered tree, each node having file_touched, acceptance, and rollback.
  • Why separate: planning and coding benefit from different prompts + different models (Opus for planning, Sonnet for coding).
# Planner prompt core
Break this spec into atomic tasks <=150 LOC each.
For each task:
  1. file(s) touched
  2. public interface (if any)
  3. acceptance test (one sentence, executable)
  4. rollback strategy
Emit as markdown, no code.
05

The Coder loop: feature branches are cheap

One task per branch. One branch per PR. One PR per review cycle. Agents get lost in big changes — humans too.

  • Agent reads the task from plan.md.
  • Writes code + tests together (TDD preferred — tests constrain hallucination).
  • Runs pnpm typecheck, pnpm test, pnpm lint locally before opening the PR.
  • PR description is auto-generated from the task node.
06

Reviewer agent in CI (the quality floor)

A GitHub Action that runs a reviewer subagent on every PR. Not a gate — a signal amplifier.

  • Loads the diff, the touched files' full contents, the spec, and the eval suite.
  • Checks: missing error paths, unbounded loops, N+1 queries, security smells (SSRF, injection), test coverage of new branches.
  • Posts inline comments prefixed 🤖 so they're easy to filter.
07

Evals: the part nobody skips and lives

Evals are for your agents, not just your product. Without them, every prompt tweak is a vibe.

  • Regression set — 20-50 real tasks with human-graded outcomes. Run nightly.
  • LLM-as-judge — a separate judge agent scores PR quality against a rubric.
  • Red-team suite — prompt-injection attempts, malicious diffs, tool-abuse scenarios.
  • Publish eval scores to a dashboard the whole team sees every Monday.
08

Release & Oncall agents (enterprise-ready path)

This is where you move from "AI-assisted shop" to "agentic org".

  • Release agent runs migrations in staging, smoke tests, canary deploys, watches SLOs, rolls back automatically on error-rate regressions.
  • Oncall agent receives PagerDuty, queries logs + traces, drafts an incident summary within 5 minutes, and suggests the three most likely causes with test commands.
  • Humans approve production writes. Always.
09

Security hardening for enterprise buyers

If your agents will touch production code or customer data, expect questions from procurement.

  • Prompt-injection defense: strip tool-use responses of instruction-like content; deny-list risky tokens; sandbox all code execution.
  • Data residency: pin model provider region; no cross-border training opt-ins.
  • Logging: every agent action logged with actor, input hash, tool invocation, output hash. 1-year retention.
  • Policy: a one-pager titled "AI in the SDLC" you can hand to a CISO. Cover data flow, human gates, incident response.
10

Team rituals that make it stick

  • Prompt review — 30 min weekly; we read changes to agent prompts like we read code reviews.
  • Skill drop — every eng who solves a tricky workflow skillifies it before end of week.
  • Eval review — Monday standup opens with eval deltas.
  • Trace of the week — pick one weird agent trace, read it together. Best learning tool we have.

Agile, evolved: Scrum & Kanban for agentic teams

Scrum and Kanban were designed for humans-as-the-bottleneck. Once half your "team" is agents running in parallel loops, most of the ceremonies break — and a few become radically more valuable. Here's how to evolve the frameworks instead of throwing them out.

scrum kanban WIP limits ceremonies estimation

// what stays

  • WIP limits — more important, not less. Agents happily run 20 branches you can't review.
  • Definition of Done — now includes evals passing + trace reviewed.
  • Retros — still the fastest learning loop. Add prompts + traces as artifacts.
  • Demos — the forcing function for "did it actually work?".

// what changes or dies

  • Story points — meaningless when an agent ships the task in 20 minutes. Kill them.
  • Daily standup — shrinks to 10 min, front-loaded with an agent-generated status.
  • Sprint length — 2 weeks → 3-5 days. The cycle time is real now.
  • Backlog grooming — becomes "spec quality review"; the spec is the work.

Evolution in six moves

A

Redefine the unit of work around spec + DAG, not ticket

In a human-only team, a ticket is a chunk one person can pick up. In an agentic team, the true unit is a spec that the planner agent can decompose into a task DAG (see Step 04 above).

  • Parent: a spec.md (PM-owned, template-enforced).
  • Children: auto-generated task nodes from the planner, each one PR-sized.
  • Board columns track the spec; a collapsible drawer shows the child tasks the agent is chewing on.
  • You stop estimating children. You only estimate specs — and only in "will this fit in one cycle?" buckets.
B

Add lanes for agent state, not just human state

Classical Kanban: Todo → In Progress → Review → Done. Agentic Kanban separates agent work from human gates.

# Agentic Kanban columns
Spec Ready   →  Planner queued  →  Agent Building  →  Agent Self-Check
             →  Human Review    →  CI + Evals      →  Canary        →  Done
  • Agent Self-Check — the agent runs tests, reviewer subagent, and eval subset before a human looks. Half the PRs get sent back here, not to the human.
  • Human Review has a strict WIP limit = number of humans × 2. Everything else queues.
  • Agent columns can run wide (5-10 in flight); human columns stay narrow.
C

WIP limits become the central governance mechanism

The single biggest failure mode of agentic engineering is humans drowning in agent output. WIP limits are how you prevent it.

  • Per human: max 3 PRs in "Human Review" queue at once. Exceed it, and the agent pauses spawning new work.
  • Per agent: max 1 active branch per agent identity. Forces sequential thinking on a single task.
  • Per repo: max N merges/day. Protects review quality and CI capacity.
  • Enforce with a board automation that flips new tasks to "blocked" when a limit is hit.
Little's Law still applies. Throughput = WIP / Cycle Time. Agents reduce cycle time dramatically — if you don't cap WIP, you just create a review backlog.
D

Ceremonies, rewritten

  • Standup (10 min, async-first) — agent posts a generated status: what merged, what's stuck in review, what's blocked. Humans only speak to flag a blocker the agent missed.
  • Planning (weekly, 30 min) — replaces sprint planning. Focus is on which specs ship this week. Decomposition is the planner agent's problem, not the team's.
  • Spec review (2×/week, 45 min) — new ceremony. Replaces backlog grooming. Humans sharpen the specs so agents don't execute ambiguity.
  • Prompt & eval review (weekly, 30 min) — we read changes to agent prompts and eval deltas like we read code. This is the meta-ceremony that keeps the system improving.
  • Demo (weekly, 20 min) — ships or it didn't happen. Includes at least one customer-facing increment.
  • Retro (every 2-3 weeks, 45 min) — three new categories: prompts that misfired, tools we wished an agent had, places we caught an agent just in time.
E

Roles, re-cast

Titles don't disappear; the job description changes.

RoleBeforeIn an agentic team
Product OwnerWrites tickets, prioritizes backlogWrites specs agents can execute; owns spec-quality ceremony
Scrum MasterRuns ceremonies, removes blockersFlow engineer: tunes WIP limits, agent prompts, CI capacity
EngineerPicks ticket, writes codeReviews, pair-programs with agent, writes hard paths & tests
Tech LeadArchitecture, code reviewOwns the planner prompts, the reviewer subagent, and the eval suite
QAWrites test casesEval engineer: owns regression sets, red-team prompts, quality dashboard
F

Metrics that actually reflect agentic flow

Velocity in story points is noise. Track these instead.

  • Spec-to-merge lead time — p50 and p90. Your real cycle time.
  • First-pass merge rate — % of agent PRs merged with ≤1 round of human feedback. Agent quality gauge.
  • Review queue depth — if p95 PR waits >24hr for a human, your WIP limit is too loose.
  • Eval pass rate — trend over time. Dropping = prompt/tool regression. Investigate same-day.
  • Escaped defects per spec — bugs found in prod divided by specs shipped. Your real quality bar.
  • Human-edit ratio — fraction of agent-authored LOC changed in review. Rising = specs are getting sloppier or prompts stale.
Bottom line: Scrum is pressure applied at the sprint boundary; Kanban is pressure applied at the WIP line. Agentic teams lean toward continuous flow with a weekly heartbeat — Kanban bones, one Scrum-style demo + retro cadence, and ceremonies rebuilt around specs and prompts instead of tickets and points.
