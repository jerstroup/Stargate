// DATA & ANALYTICS

A text-to-SQL analyst every teammate can ask

Early-stage companies don't need a data team. They need one clean warehouse, a semantic layer, and an agent that answers questions in plain English — with the SQL shown so humans can verify. This stack scales from 3 people to 50 without rewriting the analytics tier.

text-to-SQL semantic layer dbt warehouse
01

One warehouse. Pick it and commit.

Do not run analytics off your prod DB. Get a warehouse with a free tier on day one.

  • BigQuery, Snowflake, or Motherduck — all fine. BigQuery is easiest if you're Google-shop.
  • Use Fivetran/Airbyte/dltHub to pipe in: prod DB replicas, Stripe, Segment, Salesforce, Intercom.
  • Dated partitioning on every fact table. Pay-per-query means you want pruning.
02

Build the semantic layer before building dashboards

The semantic layer is the unlock for agentic data. It's how agents know what "active user" means.

  • dbt + a metrics/semantic config (dbt Metrics, Cube, or MetricFlow).
  • Every metric has: definition, grain, owner, filters, and the approved SQL.
  • This is the same metric dictionary referenced in Finance. One file. Two consumers.
# metrics.yml
metrics:
  - name: active_users_7d
    description: Distinct users who performed a core action in the last 7 days.
    grain: user
    sql: SELECT COUNT(DISTINCT user_id) FROM events WHERE action IN (...) AND ts > NOW() - INTERVAL 7 DAY
    owner: product
03

The analyst agent (text-to-SQL, done right)

The difference between a fun demo and a trusted tool is how you wire context.

  • Agent has read access to: the semantic layer, dbt model docs, past queries (successful ones), a few "gold" annotated queries.
  • For every question it: (a) proposes the SQL, (b) runs it against a sample, (c) shows the SQL + result, (d) asks if you want the full run.
  • Never phrases the answer as a fact without citing the SQL.
Guardrail: read-only role for the agent. No DELETE, no DROP, no cross-dataset joins without explicit grant.
04

Self-serve dashboards, generated from metrics

Skip Tableau at this stage. Let the analyst agent emit Lightdash/Metabase configs from the semantic layer.

  • One dashboard per role: founder KPI, sales, CS, finance, engineering SLIs.
  • The agent generates the first pass; humans adjust layout.
  • Every widget traces back to a named metric — no duplicate definitions floating around.
05

Anomaly detection (the alert you actually want)

Forget static thresholds. A nightly agent checks every top-line metric for shifts.

  • Simple model: 14-day moving baseline + 3σ envelope.
  • When it fires, the agent attempts to explain: segments that moved, correlated deploys, ticket volume shifts.
  • Posts a "here's what changed and why I think so" to Slack, with the SQL.
06

Feature-adoption agent

Per feature shipped behind a flag, the agent owns the measurement loop.

  • Defines the adoption funnel from the spec.
  • Produces a weekly cohort-retention curve.
  • Compares to the pre-registered hypothesis and reports success/fail with confidence interval.
07

Data-quality watchdog

Trust in the warehouse is everything. Treat schema changes as incidents.

  • dbt tests on every model: not-null on primary keys, unique on join keys, accepted-values on enums.
  • Freshness checks: every source table updated within its SLA.
  • When a test fails, the agent opens a ticket, identifies the upstream schema change (if any), and suggests the fix.
08

Scale path

At 20 people you'll want:

  • A reverse-ETL back into Salesforce/HubSpot/Intercom (Hightouch/Census).
  • A lightweight experimentation platform (Statsig/GrowthBook).
  • Row-level permissions on sensitive tables; the agent respects them.

At 50+ people, hire a real data lead. The agent becomes their lever.

Pitfalls

// don't

  • Let the agent run against prod DB
  • Have multiple definitions of ARR
  • Ship dashboards before the semantic layer
  • Trust SQL you can't read

// do

  • Start with 10 metrics, not 200
  • Show the SQL, always
  • Version-control everything
  • Sample-run before full-run on large queries
